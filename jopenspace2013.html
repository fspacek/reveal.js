Princip "good enough"

  • stavíme na tom, co známe
  • známe své limity
  • od jednoduššího ke složitějšímu

customizace Spring Frameworku

  • adopce externího modulového rámce
  • pohybujeme se na neznámé půdě
  • zjednodušování rámce pro vlastní použití

OSGI

Základní listina práv

  • všichni jsme si rovni
  • každý má právo na soukromí
  • musíme se umět domluvit
  • každý má své poslání
  • jeden doplňuje druhého
  • každý má právo na svého dvojníka

A přínosy?

  • rozbití monolitu a snížení provázanosti
  • zpřístupnění programování základního produktu firmy více vývojářům
  • lepší dělba práce
  • princip přirozeného výběru
  • variabilita
  • kompozice
  • možnosti zásahů do aplikace za běhu

Izolace business vrstvy

  • jednotný životní cyklus
  • oddělení "kontextů" jednotlivých modulů
  • jednotné principy komunikace
    • předávání zpráv (observer pattern)
    • přímá volání přes veřená rozhraní
  • lokalizace
  • závislosti

Závislosti

  1. zprvu neformálně
    → problém se stanovením pořadí startu modulů
  2. formalizace na explicitní závislosti
    → libovolné pořadí s využitím proxy objektů
    • povinné / nepovinné
    • deklarace konzumovaného rozhraní

Izolace datové vrstvy

  • každý modul se musí postarat o vytvoření úložných struktur.
  • dvě instance jednoho modulu musí mít svá data oddělená (skriptované DDL)
  • evoluce datového modelu - verzování (ala Liquibase)
  • možnost rozšiřování datového modelu bez narušení funkcionality modulu

Izolace UI

Modul, který poskytuje uživatelské rozhraní, by se o něj měl starat sám. Odhalovat by je měl až je o to požádán. Rozhraní musí být jednoduše modifikovatelné, protože zákazníci na modularitu nehledí.

Boj s Java platformou

  • JSP ztrácí dech, FreeMarker jediný vyhovuje
  • komplikace s většinou web frameworků
  • natahování standardů (ani Servlet 3.0 nám nestačí - start kontejneru != start modulu)
  • problém s vystavením jednoho UI ve dvou instancích

Odvrácená strana mince

  • délka inicializační fáze Spring Framework není nejrychlejší nebo toho po něm chceme hodně (dynamické proxy, autowiring)
  • náchylnost k memory leakům jeden špatně se chovající modul stačí
  • transitivní dependence daň za sdílený classloader - řešíme centralizací verzí a testy
  • ne všechny moduly mají potřebnou kvalitu ale naštěstí máme přirozený výběr :)
